# ELRS 机内 UI 功能目标 TODO

目标：在 Linux 遥控器上，恢复并稳定提供“机内 UI 可操作 ELRS 参数与 Bind”的能力，同时不影响主控发控链路可用性。

## 范围说明

- 本文只定义功能目标与验收结果。
- 不包含具体实现方案、模块拆分、代码结构或协议设计细节。

## 功能目标

1. 机内 UI 可进入 ELRS 页面并显示真实模块连接状态。
2. UI 可浏览 ELRS 参数列表与层级（含目录进入/返回）。
3. UI 可修改常用 ELRS 参数，并能看到修改结果反馈。
4. UI 可触发 Bind 操作，并能看到操作状态与结果。
5. UI 可编辑字符串类参数（如 Bind Phrase），并完成提交。
6. UI 可显示关键链路状态（至少包含连接状态、最近更新状态、基础链路质量信息）。
7. 参数操作失败时，UI 可显示可读错误信息（超时、未连接、写入失败等）。
8. 系统在无 UI 场景下仍可保持主控链路工作（UI 不是运行依赖）。

## 可用性目标

1. ELRS 页面操作路径清晰，常见操作不超过 3 层导航。
2. 参数刷新、参数写入、Bind 操作均有明确“进行中/完成/失败”状态。
3. 在模块暂时离线或串口异常时，UI 不崩溃、不假死，并可恢复。

## 兼容性目标

1. 与当前主链路目标一致：`输入 -> mixer -> 发射` 不中断。
2. 与现有验证文档可对齐，形成可重复执行的板级验证步骤。
3. 后续可扩展更多 ELRS 参数与状态字段，不要求一次性覆盖全部。

## 验收标准

1. 能在板端 UI 上完成一次完整流程：
参数浏览 -> 修改一个参数 -> 提交成功 -> 触发 Bind（或收到明确失败原因）。
2. 主输入链与发射链在 ELRS 页面操作期间保持可用，无明显卡顿或中断。
3. 断开模块、错误串口、写入失败三种场景下，UI 均能给出可读错误反馈。
4. 有对应的验证记录（命令、日志路径、结果）可复现。

## 非目标（当前阶段）

1. 不要求完整覆盖所有 ELRS 厂商自定义参数。
2. 不要求实现高级自动化脚本编排。
3. 不要求在本阶段完成全部遥测字段可视化。
