# LinTx 航模遥控总 TODO

目标：完成一个可实际使用的 Linux 航模遥控系统，覆盖输入采集、模型配置、混控、ELRS/CRSF 射频链路、遥测、命令行控制、GUI 和整机验证。

## 整体架构

我建议后续按下面这套固定架构推进，不再继续堆零散模块。

核心原则：

- `headless first`
- 核心遥控链不依赖 GUI
- GUI 和 CLI 都只是控制面
- 没有 GUI 时，系统仍可完成启动、配置、运行、监视和验证

### 1. 输入层

统一输入服务：

- `input_service`
- 负责接入 `ADC`、`STM32 Serial`、`CRSF RC In`、`joydev`、`mock`
- 统一输出标准化 `InputFrame`

### 2. 模型层

统一模型服务：

- `model_service`
- 负责加载 `radio.toml` 和 `models/*.toml`
- 负责活动模型、输入映射、profile、输出协议、failsafe

### 3. 控制层

统一混控服务：

- `control_service`
- 负责 `input -> normalize -> expo/curve -> mix -> output`
- 输出标准化 `OutputFrame`

### 4. 射频链路层

统一 ELRS/CRSF 链路服务：

- `rf_link_service`
- 独占 UART
- 同时负责 RC 发射、ELRS 参数树、链路状态、遥测解析
- 不再保留 `elrs_tx` 和 `elrs_agent` 分裂持串口的方式

### 5. 遥测层

统一遥测服务：

- `telemetry_service`
- 汇总 RF 链路回传、系统状态、外部传感器状态

### 6. 控制面层

统一控制面：

- `ui_service`
- `cli_service`
- 两者都只读写统一后端消息
- 两者都不能绕过核心服务直接拼业务逻辑

说明：

- GUI 是可选前端
- CLI 是一等公民
- 系统必须支持无界面运行

### 7. 运行层

统一运行模式：

- `runtime_service`
- 负责启动整机核心服务
- 支持 `headless` 模式
- 支持 `gui` 模式
- 支持 `cli attach` 或命令式操作

### 8. 验证层

统一验证体系：

- `validation`
- 覆盖 host mock、板级联调、整机链路验证

## 功能单元 TODO

下面按“功能单元”划分，每一项都是一个完整能力，而不是碎任务。

### 功能单元 A：输入系统

目标：

- 所有输入源都走统一接入链，形成标准 `InputFrame`

包含内容：

- ADC 摇杆输入
- STM32 串口输入
- CRSF RC 输入
- joydev 输入
- mock 输入

要达到的结果：

- 输入源可切换
- 输入数据可监视
- 输入源状态可见
- 不再把所有输入都硬塞进 4 通道 ADC 语义

完成标志：

- GUI 或 CLI 能看到当前输入源和通道值
- 运行链不再依赖 `AdcRawMsg { value: [i16; 4] }`

### 功能单元 B：校准与输入预处理系统

目标：

- 把 ADC 校准和数字输入预处理分开

包含内容：

- ADC 校准
- 数字输入归一化
- 输入反向
- 死区、范围、中心点处理

要达到的结果：

- ADC 需要校准
- CRSF/STM32 数字输入不依赖 `joystick.toml`

完成标志：

- 切换输入源后，系统行为合理，不会因为校准文件缺失而整条链不可用

### 功能单元 C：模型与配置系统

目标：

- 所有遥控行为由模型配置驱动

包含内容：

- 活动模型管理
- 输入映射
- profile
- 输出协议
- 通道顺序
- failsafe
- 基础系统设置

要达到的结果：

- 切模型会真实改变运行链行为
- GUI 和 CLI 都可切活动模型
- 系统设置能持久化

完成标志：

- 当前模型、profile、输入映射、输出协议都来自配置文件而不是硬编码

### 功能单元 D：混控与控制系统

目标：

- 建成最小可用航模混控链

包含内容：

- normalize
- expo
- curve
- mix
- output limit
- rate profile

要达到的结果：

- 固定翼、四轴、车模等能基于同一控制链处理
- 输出结果受模型参数真实控制

完成标志：

- 输入变化、模型切换、profile 切换都会反映到最终输出

### 功能单元 E：ELRS / CRSF 射频链路系统

目标：

- 建成单 UART、单服务的完整 RF 链路

包含内容：

- RC 发射
- ELRS 参数树
- 连接状态
- Bind / Bind Phrase
- Packet Rate
- Telemetry Ratio
- TX Power
- WiFi Update

要达到的结果：

- GUI 和 CLI 都可直接配置 ELRS
- RC 发射和参数配置不再冲突

完成标志：

- `rf_link_service` 独占 UART
- GUI 和 CLI 至少有一种可完成连接、浏览参数、修改参数、对频

### 功能单元 F：遥测与状态系统

目标：

- 让 GUI 和 CLI 看到的链路、电池、信号状态来自真实后端

包含内容：

- RSSI
- LQ
- RF Mode
- TX Power
- 接收机/遥控器电池
- 链路状态
- 最近更新时间

要达到的结果：

- 去掉核心链路上的 `system_state_mock`
- GUI 能显示真实链路状态

完成标志：

- Telemetry 页面、CLI 状态输出都不再依赖 mock 定时器

### 功能单元 G：命令行控制与无界面运行系统

目标：

- 让系统在没有 GUI 时也能完整工作

包含内容：

- headless 启动
- CLI 配置查询
- CLI 配置修改
- CLI 模型切换
- CLI ELRS 操作
- CLI 遥测查看
- CLI 验证入口

要达到的结果：

- 用户可以只用命令行完成日常操作
- GUI 只是更方便的入口，不是系统依赖

完成标志：

- 可以在纯命令行模式下完成启动、模型切换、ELRS 连接/配置、状态查看、验证

### 功能单元 H：GUI 遥控器系统

目标：

- 把当前 launcher/demo UI 重构成真正的遥控器 GUI，但不承担核心运行依赖

包含内容：

- Home
- Models
- Inputs
- Mixer
- Outputs
- ELRS
- Telemetry
- System

要达到的结果：

- 页面名称和能力一致
- 所有核心页面都接真实后端
- 不再保留主流程中的模板页和伪页面

完成标志：

- 能在 GUI 上完成日常遥控操作与状态查看

### 功能单元 I：STM32 集成系统

目标：

- 实现“GUI 控制 ELRS，并自动读取 STM32 内容”的整机体验

包含内容：

- STM32 协议定义
- STM32 输入状态显示
- STM32 在线状态
- 与输入系统联通

要达到的结果：

- STM32 变成正式输入源，而不是实验型串口模块

完成标志：

- GUI 或 CLI 能直接确认 STM32 是否在线、通道是否更新

### 功能单元 J：输出系统

目标：

- 建立统一输出层，而不是每个输出驱动各自吃旧结构

包含内容：

- CRSF / ELRS 输出
- USB HID 输出
- 输出通道顺序
- failsafe 行为

要达到的结果：

- 输出协议和模型配置一致
- 输出结果可监视

完成标志：

- `OutputFrame` 成为统一输出接口

### 功能单元 K：整机验证系统

目标：

- 让“航模基本功能验证”成为固定流程

包含内容：

- host mock 验证
- 输入链验证
- ELRS GUI 验证
- ELRS CLI 验证
- STM32 验证
- 整机链路验证

要达到的结果：

- 每个功能单元都有标准验证方法
- 不再靠口头记忆测试流程

完成标志：

- 有固定脚本、固定步骤、固定验收结果

## 开发顺序

建议按下面顺序推进：

1. 输入系统
2. 校准与输入预处理系统
3. 模型与配置系统
4. 混控与控制系统
5. ELRS / CRSF 射频链路系统
6. 遥测与状态系统
7. 命令行控制与无界面运行系统
8. GUI 遥控器系统
9. STM32 集成系统
10. 输出系统
11. 整机验证系统

## 最终验收目标

最终不是“某几个模块能跑”，而是达到下面这个整机结果：

- 系统可在 `headless` 模式启动
- 系统可在 `gui` 模式启动
- 命令行可完成模型选择、状态查看、ELRS 连接与关键配置
- GUI 可完成同样的日常操作，但不是运行依赖
- 可查看输入
- 可完成 ADC 校准或确认 STM32/CRSF 输入在线
- 可自动读取 STM32 内容并参与控制链
- 可稳定输出 RC 数据
- 可显示真实遥测与链路状态
- 有固定验证流程支撑日常开发
